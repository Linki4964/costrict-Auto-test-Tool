1. 针对“一次性成功率”的稳定性指标由于你要求脚本必须先登录，登录环节的稳定性直接决定了“一次性”的成败。
    - 指令对齐稳定性 (Prompt-Instruction Alignment): 衡量 AI 是否每次都能严格遵循格式要求。
        - 量化方式： 统计 $100$ 次执行中，有多少次因为 AI 输出格式错误（如多写了注释、JSON 格式错乱）导致代码无法直接运行。
2. 针对“自动化程度”的稳定性指标自动化程度高意味着“异常处理”不需要人工干预。
    - 自愈成功率 (Self-Healing Success Rate): 当 UI 发生微调（如 ID 变化但文本未变）时，AI 能够自动修正定位符并执行成功的概率。
    - 人工介入次数: 工作流在开始前自动检查登录状态、网络环境、权限配置的准确性，减少因环境问题导致的人工介入。
3. 针对“执行效率”的稳定性指标调用次数越少，不仅成本越低，出错的概率（受网络波动影响）也越小。
    - Token 消耗波动率 (Token Usage Variance): 针对同类复杂度的用例，统计 AI 消耗 Token 的标准差。波动越小，说明 AI 的思考逻辑越趋于稳定，没有产生无效的循环或冗余尝试。
    - 平均尝试次数 (Mean Attempts to Success): 达成预期结果平均需要的 LLM 调用轮数。如果该数值稳定在 $1.1 - 1.2$ 之间，说明效率极高且稳定。
4. 针对“适配程度”与“覆盖度”的稳定性指标
    - 跨项目执行通过率 (Cross-Project Pass Rate): 在不同技术栈（如 React、Vue 或原生 App）上应用同一工作流，其“一次性成功率”的衰减情况。
    - 衰减越小，适配稳定性越好。覆盖分布标准差 (Coverage Distribution Variance): 针对同一模块多次生成用例，统计其覆盖的逻辑路径是否一致。如果每次生成的用例覆盖点完全随机，说明工作流对“覆盖度”的控制力不足。
